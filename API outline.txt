### **API Architecture: FastAPI Backend**

The backend will be built using FastAPI, leveraging its strengths in performance, automatic documentation (via OpenAPI/Swagger), and data validation with Pydantic. The API will be versioned (`/api/v1/`) and will follow RESTful principles for data retrieval, while utilizing WebSockets for real-time communication.

**Core Principles:**

1.  **Instrument-Centric:** Most endpoints will be nested under `/instruments/{instrument_name}/` to align with the UI's global state.
2.  **Asynchronous Operations:** Long-running tasks (like starting a pipeline) will be handled asynchronously, immediately returning a task ID to the client, with progress communicated via WebSockets.
3.  **Read-Only Data Access:** The primary function is to read and serve data from the Parquet files generated by the pipeline. The API will not modify the results of a completed run.
4.  **Data Transformation:** The API is responsible for performing necessary data aggregations and transformations (e.g., filtering, pivoting, calculating histogram bins) to send UI-ready data to the frontend, minimizing client-side processing.

### **API Endpoint Definitions**

#### **1. Global & Application State Endpoints**

These endpoints provide the initial data needed to populate the UI shell.

**`GET /api/v1/instruments`**
*   **Purpose:** To populate the global Master Instrument selector.
*   **Data Source:** Scans the `raw_data/` directory.
*   **Processing:** Lists all `.csv` files and extracts the instrument names.
*   **Returns (JSON):** A list of available instrument strings.
    ```json
    {
      "instruments": ["EURUSD60", "XAUUSD240", "GBPUSD30"]
    }
    ```

**`GET /api/v1/status`**
*   **Purpose:** To get a quick overview of all currently running pipeline processes system-wide.
*   **Data Source:** An in-memory state manager in the FastAPI application that tracks active subprocesses.
*   **Processing:** Returns a list of active runs.
*   **Returns (JSON):**
    ```json
    {
      "active_runs": [
        { "run_id": "uuid-123", "instrument": "EURUSD60", "stage": "platinum_discoverer" }
      ]
    }
    ```

#### **2. Dashboard Endpoints (Tab 1)**

**`GET /api/v1/instruments/{instrument_name}/dashboard`**
*   **Purpose:** To provide all the necessary KPI data for the Mission Control tab for the selected instrument.
*   **Data Sources:** `diamond_data/master_reports/`, `bronze_data/`, `platinum_data/combinations/`, `platinum_data/discovered_strategies/`.
*   **Processing:**
    1.  Reads the `master_reports/{instrument_name}.parquet` file to get the count of master strategies.
    2.  Reads the metadata (or row counts) from the relevant files in other layers to get the numbers for the discovery funnel.
*   **Returns (JSON):**
    ```json
    {
      "masterStrategyCount": 22,
      "discoveryFunnel": {
        "bronzePossibilities": 1250000000,
        "platinumBlueprints": 45000,
        "platinumStrategies": 8500,
        "diamondMasters": 22
      }
    }
    ```
*   **Note:** If any required file is missing, it will return an appropriate error or null values, which the UI can interpret to show that a stage hasn't been run.

#### **3. Strategy Explorer Endpoints (Tab 2)**

**`GET /api/v1/instruments/{instrument_name}/strategies/filters`**
*   **Purpose:** To populate the filter dropdowns with available options.
*   **Data Source:** `platinum_data/combinations/{instrument_name}.parquet`.
*   **Processing:** Reads the combinations file, gets the unique values from the `sl_def`, `tp_def`, and `trade_type` columns.
*   **Returns (JSON):**
    ```json
    {
      "tradeTypes": ["buy", "sell"],
      "slDefinitions": ["ratio", "support", "resistance", "SMA_50"],
      "tpDefinitions": ["ratio", "resistance", "BB_upper_20"]
    }
    ```

**`GET /api/v1/instruments/{instrument_name}/strategies`**
*   **Purpose:** This is the core endpoint for the screener. It fetches a list of master strategies based on a complex set of query parameters.
*   **Query Parameters:** `?pf_min=1.5&sqn_min=1.8&trade_type=buy&min_markets_passed=3...`
*   **Data Sources:** `diamond_data/master_reports/{instrument_name}.parquet`, `diamond_data/final_reports/{instrument_name}_summary.parquet`, and the individual trade logs in `diamond_data/trade_logs/` for the sparkline.
*   **Processing:**
    1.  Loads the master reports and summary reports DataFrames.
    2.  Merges them on `trigger_key`.
    3.  Applies all filters from the query parameters using Pandas filtering.
    4.  For each filtered strategy, it reads the corresponding trade log for the master instrument to compute the cumulative PnL for the equity sparkline.
    5.  Paginates the results.
*   **Returns (JSON):**
    ```json
    {
      "strategies": [
        {
          "triggerKey": "a1b2c3d4e5f6",
          "tradeType": "buy",
          "avgProfitFactor": 2.15,
          "avgWinRate": 62.1,
          "marketsPassed": 4,
          "equityCurve": [10000, 10050, 10020, 10150, ...]
        }
      ],
      "totalCount": 22
    }
    ```

#### **4. Strategy Analysis Workbench Endpoints (Tab 3)**

These endpoints all relate to a single strategy, identified by its `trigger_key`.

**`GET /api/v1/instruments/{instrument_name}/strategies/{trigger_key}/master_backtest`**
*   **Purpose:** Fetches detailed in-sample backtest results and robustness check data.
*   **Data Sources:** `diamond_data/master_reports/{instrument_name}.parquet` and the corresponding trade log from `diamond_data/trade_logs/`.
*   **Processing:**
    1.  Filters the master report for the specific `trigger_key` to get all summary metrics.
    2.  Reads the trade log to get the full PnL series for the equity curve chart.
    3.  **On-the-fly:** Runs Monte Carlo simulations (shuffling trades, etc.) on the PnL series to generate the data for the robustness histogram.
*   **Returns (JSON):**
    ```json
    {
      "metrics": { "profitFactor": 2.8, "maxDrawdown": 1250.5, ... },
      "equityCurve": { "timestamps": [...], "values": [...] },
      "monteCarlo": { "finalEquityDistribution": [...] }
    }
    ```

**`GET /api/v1/instruments/{instrument_name}/strategies/{trigger_key}/gauntlet`**
*   **Purpose:** Fetches the cross-market validation results for the heatmap.
*   **Data Source:** `diamond_data/final_reports/{instrument_name}_detailed.parquet`.
*   **Processing:** Reads the file and filters for the specific `trigger_key`.
*   **Returns (JSON):** An array where each object represents a validation market.
    ```json
    [
      { "market": "EURUSD60", "isMaster": true, "profitFactor": 2.8, "winRate": 68.2, ... },
      { "market": "GBPUSD60", "isMaster": false, "profitFactor": 1.9, "winRate": 61.5, ... }
    ]
    ```

**`POST /api/v1/instruments/{instrument_name}/strategies/{trigger_key}/pivot`**
*   **Purpose:** Provides the data for the flexible Regime Pivot Analysis table. It's a `POST` request because the query structure is complex and sent in the body.
*   **Request Body (JSON):**
    ```json
    {
      "rows": ["session"],
      "columns": ["market"],
      "metric": "profitFactor"
    }
    ```
*   **Data Source:** `diamond_data/final_reports/{instrument_name}_regime_analysis.parquet`.
*   **Processing:**
    1.  Reads the regime analysis file and filters for the specific `trigger_key`.
    2.  Uses Pandas' `pivot_table()` function, passing the parameters from the request body directly.
    3.  Formats the resulting pivot table into a JSON structure that's easy for the UI to render.
*   **Returns (JSON):**
    ```json
    {
      "columns": ["EURUSD60", "GBPUSD60"],
      "index": ["London", "NewYork", "Tokyo"],
      "data": [
        [2.9, 1.8],
        [2.5, 2.1],
        [0.8, 0.9]
      ]
    }
    ```

**`GET /api/v1/instruments/{instrument_name}/strategies/{trigger_key}/visualizer`**
*   **Purpose:** To fetch the OHLC and trade data needed for the interactive chart.
*   **Query Parameters:** `?market=GBPUSD60&start_date=YYYY-MM-DD&end_date=YYYY-MM-DD`
*   **Data Sources:** `silver_data/features/{market}.parquet` and `diamond_data/trade_logs/{instrument_name}/{market}/{trigger_key}.parquet`.
*   **Processing:**
    1.  Reads the OHLC data from the Silver features file for the requested `market`.
    2.  Reads the specific trade log for the strategy on that market.
    3.  Filters both datasets by the date range.
*   **Returns (JSON):**
    ```json
    {
      "ohlc": [ { "time": "...", "open": 1.2, "high": 1.3, "low": 1.1, "close": 1.25 } ],
      "trades": [ { "entry_time": "...", "exit_time": "...", "pnl": 50.1, "sl_price": 1.15, "tp_price": 1.35 } ]
    }
    ```

#### **5. Pipeline Manager & Real-time Endpoints (Tab 4)**

**`POST /api/v1/pipeline/run`**
*   **Purpose:** To initiate a new pipeline run. This is an asynchronous endpoint.
*   **Request Body (JSON):**
    ```json
    {
      "instrument": "EURUSD60",
      "runType": "full_pipeline",
      "configProfile": "default"
    }
    ```
*   **Processing:**
    1.  Validates the request.
    2.  Generates a unique `run_id`.
    3.  Spawns the `orchestrator.py` script as a background subprocess, passing the necessary arguments.
    4.  Immediately returns a success response.
*   **Returns (JSON):**
    ```json
    {
      "message": "Pipeline run successfully initiated for EURUSD60",
      "run_id": "uuid-1234"
    }
    ```

**`POST /api/v1/pipeline/stop`**
*   **Purpose:** To stop a currently running pipeline process.
*   **Request Body (JSON):** `{ "run_id": "uuid-1234" }`
*   **Processing:** Finds the subprocess associated with the `run_id` and terminates it.

**`WS /ws/v1/pipeline/logs/{run_id}`**
*   **Purpose:** A WebSocket endpoint for streaming the live console output of a running pipeline directly to the UI.
*   **Protocol:**
    1.  React UI initiates a WebSocket connection to this endpoint after receiving a `run_id` from the `POST /run` call.
    2.  The FastAPI backend captures the `stdout` of the running subprocess line-by-line.
    3.  Each captured line is sent as a text message over the WebSocket to the client.
    4.  When the process finishes, a special message like `{"event": "complete", "status": "success"}` is sent, and the connection is closed.
