# Silver Layer: The Enrichment Engine

The script `silver_data_generator.py` is the central feature engineering hub of the strategy discovery pipeline. It transforms the raw, high-volume trade simulations from the Bronze Layer into an intelligent, context-rich dataset ready for analysis and machine learning.

This layer has a crucial dual role, operating in two distinct stages for each financial instrument.

## Core Purpose

The Silver Layer bridges the gap between raw price data and abstract strategic concepts. Its purpose is to create a "fingerprint" of the market's state at any given moment and then use that fingerprint to describe _why_ a potential trade might have been good. It answers questions like, "Was this winning trade's stop-loss placed just below a key moving average?" or "Did this trade occur during a low-volatility, ranging market in the Tokyo session?". This contextual enrichment is what allows the Platinum Layer to discover meaningful, human-readable trading rules.

## How It Works

The script operates in two major sequential stages.

### Stage 1: Market Feature Generation

First, the script focuses on understanding the market itself, independent of any trades.

1.  **Data Ingestion**: It consumes the raw OHLC price data from a `.csv` file in the `raw_data/` directory.
2.  **Comprehensive Feature Calculation**: It calculates a vast suite of over 200 market features for every single candle, including:
    - **Standard Indicators**: Moving Averages (SMA, EMA), Bollinger Bands, RSI, MACD, ATR, ADX, and more, using multiple time periods for each.
    - **Candlestick Patterns**: It uses the `talib` library to recognize dozens of classic candlestick patterns (e.g., Doji, Hammer, Engulfing patterns), scoring each candle for its pattern match.
    - **Support & Resistance**: It identifies fractal support and resistance points using a fast, Numba-accelerated algorithm and forward-fills them to provide S/R levels for every candle.
    - **Market Context Features**: It calculates higher-level concepts like:
      - **Trading Sessions**: Categorizes each candle into 'Tokyo', 'London', 'New_York', and their critical overlap periods.
      - **Market Regimes**: Classifies the market as 'trending' or 'ranging' (based on ADX) and 'high-volatility' or 'low-volatility' (based on ATR).
      - **Price Action Stats**: Calculates rolling metrics like the ratio of bullish to bearish candles, average candle body size, and average candle range over various lookback periods.
3.  **Output**: The result of this stage is a single, wide **Parquet** file saved to `silver_data/features/`. This file serves as the complete, definitive source of market context for the instrument.

### Stage 2: Trade Enrichment

Next, the script enriches the millions of "winning possibilities" generated by the Bronze Layer.

1.  **Data Ingestion**: It reads the enormous `bronze_data/{instrument}.parquet` file.
2.  **High-Performance Enrichment**: To avoid memory-intensive database-style merges, the script uses a highly efficient lookup method:
    - The market features from Stage 1 are loaded into memory as a lean NumPy array and a timestamp-to-index `pd.Series`.
    - The Bronze data is read in parallel chunks.
    - For each trade in a chunk, the script performs a near-instantaneous lookup using the trade's `entry_time` to fetch the complete market context from the NumPy array.
3.  **Relational Feature Calculation**: For each trade, it calculates a new set of "relational positioning" features. These describe where the SL and TP levels were placed _relative to_ key market structures. Examples include:
    - `sl_dist_to_SMA_50_bps`: "The stop-loss was X basis points away from the 50-period SMA."
    - `tp_place_pct_to_resistance`: "The take-profit was placed 80% of the way from the entry price to the nearest resistance level."
4.  **Chunked Output**: The final, fully enriched trade data is saved in a series of smaller Parquet chunks to a dedicated directory: `silver_data/chunked_outcomes/{instrument}/`. This chunked format is essential for enabling parallel processing in the subsequent Platinum layer.

## Key Architectural Features

- **Dual-Functionality**: Intelligently separates the one-time task of feature generation from the parallelizable task of trade enrichment.
- **Performance-Optimized Lookups**: Avoids slow DataFrame merges by using pre-computed NumPy arrays and Pandas Series lookups, which is orders of magnitude faster and uses significantly less memory.
- **Producer-Consumer Architecture**: For trade enrichment, the main process reads batches of Bronze data (producing tasks) which are consumed and processed in parallel by a pool of worker processes.
- **Memory Safety**: By processing the massive Bronze file in manageable batches and writing enriched data out in chunks, the script can handle datasets that are far larger than the available system RAM.
- **Parquet I/O**: The use of Parquet for both the features and the chunked outcomes ensures fast reading and writing and efficient storage on disk.

## Inputs and Outputs

- **Inputs**:

  1.  `raw_data/{instrument}.csv`: For generating market features.
  2.  `bronze_data/{instrument}.parquet`: The source of potential trades to be enriched.

- **Outputs**:
  1.  `silver_data/features/{instrument}.parquet`: A single file containing the comprehensive market features for every candle.
      - **Schema**: `time`, `open`, `high`, `low`, `close`, `SMA_20`, `RSI_14`, `CDLDOJI`, `support`, `session`, etc. (200+ columns).
  2.  `silver_data/chunked_outcomes/{instrument}/`: A directory containing numerous Parquet files (`chunk_1.parquet`, `chunk_2.parquet`, etc.).
      - **Schema**: All columns from the Bronze file, plus the new relational features (`sl_dist_to_...`, `tp_place_pct_to_...`, etc.).

## Configuration

The behavior of the Silver layer is controlled by parameters in `config.py`.

- `SILVER_INDICATOR_WARMUP_PERIOD`: The number of initial candles to discard to ensure indicators like long-period moving averages are stable.
- `SILVER_PARQUET_BATCH_SIZE`: The number of rows to read from the Bronze Parquet file in each batch during enrichment.
- **Technical Indicator Parameters**: A series of lists and variables (`SMA_PERIODS`, `EMA_PERIODS`, `MACD_FAST`, etc.) that define the entire feature engineering space.
- `PIVOT_WINDOW`: The lookback window for identifying fractal support and resistance points.
- `PAST_LOOKBACKS`: The rolling window sizes for calculating price action features.

## How to Run the Script

1.  **Via the Master Orchestrator (Recommended)**:
    The `orchestrator.py` script will run this layer automatically after the Bronze layer is complete.

    ```bash
    python orchestrator.py
    ```

2.  **Standalone (Interactive Mode)**:
    Run the script directly. It will scan for new `.parquet` files in `bronze_data` and present an interactive menu to choose which instrument(s) to process.

    ```bash
    python scripts/silver_data_generator.py
    ```

3.  **Standalone (Targeted Mode)**:
    Run the script with a specific Bronze Parquet filename as an argument.

    ```bash
    python scripts/silver_data_generator.py EURUSD60.parquet
    ```

4.  **Features-Only Mode**:
    To generate only the `silver_data/features/` file without processing Bronze data, use the `--features-only` flag with a target raw `.csv` file. This is used internally by the Diamond layer prepper.
    ```bash
    python scripts/silver_data_generator.py EURUSD60.csv --features-only
    ```
